# Процесс: AI для быстрого тестирования продуктовых гипотез (vibe-coded продукт)
## 1. Цель

- **Задача**: использовать AI для быстрого тестирования продуктовых гипотез в живом кликабельном продукте, сохраняя контроль над дизайн‑системой и UX‑качеством.
- **Принцип**: AI — инструмент ускорения, не источник истины.
- **Поток работы**:
  - Product idea → Vibe‑coded mockup → Decision: keep / discard → Figma (designer formalizes accepted parts)

## 2. Инструменты

- **Figma**: дизайн, дизайн‑система, зафиксированные UX‑решения.
- **Cursor + GitHub**: кликабельный продукт и эксперименты.
- **AI**: генерация и модификация решений.

## 3. Source of truth

### 3.1 Figma

- **Дизайн‑система**
- **Утверждённые экраны и флоу**

### 3.2 GitHub

- **`main`**: текущее стабильное состояние продукта.
- **`feature/*` / `playground/*`**: гипотезы, эксперименты, альтернативные флоу.

### 3.3 Важное ограничение

- **Нет автоматической синхронизации** между Figma и кодом.

## 4. Базовый принцип

- **Вайбкод‑продукт** используется для генерации продуктовых гипотез.
- **Результат работы продакта** в вайбкод‑продукте является экспериментальным продуктовым артефактом и **не считается**:
  - дизайном
  - UX‑решением
  - дизайн‑спецификацией
- **Figma** используется **только** для фиксации принятых решений.
- **Неуспешные эксперименты** никогда не попадают в Figma.

## 5. Как добавляются новые экраны

### 5.1 Старт

- **Экран** описывается текстом и реализуется в Cursor.
- Экран должен:
  - **использовать существующие компоненты**
  - **соответствовать дизайн‑системе**

### 5.2 Реализация

- Экран собирается в коде **из существующих компонентов**.
- **Дизайн‑система в коде** не меняется без согласования.

### 5.3 Прототипы и мокапы

- **Прототипы = мокапы**: все прототипы считаются мокапами.
- **Требования к мокапам**:
  - черно‑белые
  - один шрифт, базовый на всех сущностях
  - без активных действий внутри продукта
- **Процесс работы**:
  1. Согласуем позиции и содержание (продакт + команда) в мокапе.
  2. **Копирайтер** привлекается на этапе мокапа — каждое слово должно быть на своем месте уже на этом моменте.
  3. Отдаем дизайнеру мокап — он делает черновой макет.
  4. Когда окнули с дизайнером черновой макет, он создает финальный макет.
  5. Отдаем в разработку.

## 6. Как продакт работает с продуктом

### 6.1 Продакт может

- менять тексты
- перестраивать порядок блоков
- добавлять / убирать секции
- создавать новые страницы
- тестировать альтернативные флоу

### 6.2 Продакт не может

- создавать новые UI‑компоненты
- менять дизайн‑систему
- «перегенерировать весь продукт»
- править `main` напрямую

## 7. Версии и гипотезы

- **`main`** — стабильная версия продукта.
- **Каждая гипотеза**:
  - отдельная git‑ветка
  - отдельная цель
- **Допускается** несколько параллельных гипотез.
- **Merge в `main`** — только для принятых гипотез, которые будут переноситься в Figma.
- **Ветки с непринятыми гипотезами**:
  - не поддерживаются
  - подлежат удалению после принятия решения
  - контекст решения фиксируется в PR или issue при необходимости

### 7.1 Рекомендуемый нейминг веток

Формат:

```text
<area>--<what-is-tested>
```

Примеры:

- `onboarding--simplified-flow`
- `pricing--3-tier-structure`
- `dashboard--ai-summary`
- `checkout--single-step`

## 8. Обновления при изменениях в Figma

Если флоу меняется в Figma:

- дизайнер фиксирует изменения
- изменения описательно передаются в Cursor
- обновляются **только затронутые части** продукта
- **полная пересборка продукта** не используется

## 9. Дизайн‑решения

### 9.1 Работа с Figma

В Figma возвращается:

- новая структура экрана
- зафиксированный флоу
- новые состояния
- изменения иерархии
- новые компоненты (если приняты в систему)

В Figma не возвращается:

- A/B тексты
- временные CTA
- микроэксперименты
- технические правки

Дизайнер подключается, если:

- гипотеза признана успешной
- решение должно стать частью продукта

### 9.2 НЕ считается дизайном

- вайбкод‑страницы
- тексты, созданные для теста
- временные экраны
- гипотезы без решения

## 10. Роль AI

AI:

- предлагает варианты
- ускоряет реализацию
- помогает экспериментировать

AI не:

- принимает финальные решения
- управляет дизайн‑системой
- определяет UX‑стандарты

## 11. Антипаттерны (запрещено)

- пошагово генерировать весь продукт из Figma фрейм за фреймом
- использовать AI‑генерацию как source of truth
- смешивать эксперименты и стабильную версию

## 12. Критерий успеха

- продакт может самостоятельно тестировать гипотезы
- дизайн‑система не деградирует
- продукт остаётся управляемым
- решения фиксируются, а не теряются
